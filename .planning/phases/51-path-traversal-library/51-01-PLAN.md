<objective>
Fix path traversal vulnerabilities in two routes: the library `view` action (base64-decoded path not boundary-checked against LIBRARY_PATH) and the `files/read` route (takes any raw absolute path, zero restriction).

Purpose: Both routes serve file content to the client and accept user-controlled input for the path. Without boundary checks, an attacker can read arbitrary files on the server (e.g. `/etc/passwd`, private keys, env files). Phase 50 validated agent IDs; this phase locks down the remaining two filesystem-reading routes.
Output: `app/api/library/route.ts` — 403 for paths outside LIBRARY_PATH. `app/api/files/read/route.ts` — 403 for paths outside allowed root directories.
</objective>

  <name>Task 1: Fix library route view action — boundary check against LIBRARY_PATH</name>
  <files>app/api/library/route.ts</files>
  <action>
In the `action === 'view'` block, after building `const candidate = path.join(ENV.LIBRARY_PATH, decoded)`, add a boundary check BEFORE the `existsSync(candidate)` call:

```ts
if (!candidate.startsWith(ENV.LIBRARY_PATH + path.sep) && candidate !== ENV.LIBRARY_PATH) {
  return NextResponse.json({ success: false, error: 'Forbidden' }, { status: 403 });
}
```

**Why `path.sep` matters:** `LIBRARY_PATH + path.sep` prevents a path like `~/mission-control/library-evil/...` from passing — the directory name must actually be under LIBRARY_PATH, not just start with the same string. Check: `candidate.startsWith(ENV.LIBRARY_PATH + path.sep)` (covers all paths under the dir) OR `candidate === ENV.LIBRARY_PATH` (the root itself, unlikely to be useful but safe to allow).

The legacy DB fallback (`file.path`) is already from the database (trusted data), not user input, so it does NOT need this check. Only the base64-decoded path from the query param is attacker-controlled.

Import `path` is already imported as `import * as path from 'path'` — no new import needed.

<task type="auto">
  <name>Task 2: Fix files/read route — restrict to allowlisted root directories</name>
  <files>app/api/files/read/route.ts</files>
  <action>
The `GET /api/files/read?path=...` route accepts any absolute path from the query string and reads it. This is used by the project file browser and agent workspace viewer, so we can't remove it — but we must restrict which directories it can serve.

Add an allowlist check after extracting `filePath` and before the `existsSync` check:

```ts
import { ENV } from '@/lib/env';
import { homedir } from 'os';

// Allowed roots: library, memory vault, agent workspaces, project root
const ALLOWED_ROOTS = [
  ENV.LIBRARY_PATH,
  ENV.VAULT_PATH,
  `${homedir()}/mission-control/agents`,
  process.cwd(),
];

function isAllowedPath(p: string): boolean {
  return ALLOWED_ROOTS.some(root => p.startsWith(root + '/') || p === root);
}
```

Then add the guard immediately after extracting `filePath`:
```ts
if (!isAllowedPath(filePath)) {
  return NextResponse.json({ error: 'Forbidden' }, { status: 403 });
}
```

**Important:** Use `root + '/'` (not `path.sep`) since `filePath` will always be an absolute POSIX path on macOS/Linux. Do NOT use `path.normalize` on untrusted input before the check (normalize can collapse `../` sequences but is not a substitute for a startsWith boundary check).

**Why these 4 roots:**
- `ENV.LIBRARY_PATH` — agent output files, docs, library content
- `ENV.VAULT_PATH` — memory vault (Obsidian markdown files)
- `~/mission-control/agents` — agent workspace files (SOUL.md, MEMORY.md, etc.)
- `process.cwd()` — project source code files (for code viewer in agent context)

Do NOT allow arbitrary paths, `/tmp`, `/etc`, `/Users/kevin.macarthur` root, etc.

Import changes needed: add `import { ENV } from '@/lib/env';` and `import { homedir } from 'os';` to the existing imports.

  </verify>
