  Migrate all 6 notification-settings IPC handlers from shell-exec sqlite3 to prepare() queries.
  The `prepare` function is already imported from `./database` in main.ts.

  Handlers to migrate (approximate line numbers from verification):
  1. `notification-settings:get` (~line 1035): SELECT with session_key param
  2. `notification-settings:set` (~line 1058): CHECK existing + conditional UPDATE or INSERT
  3. `notification-settings:delete` (~line 1149): DELETE with session_key param
  4. `notification-settings:getEffective` (~line 1238): SELECT with session_key param

  Note: `notification-settings:mute` and `notification-settings:unmute` were already migrated in 01-02.

  Pattern for each handler:
  - Remove the `exec()`/`execSync()` call and the `sqlite3 "${froggoDbPath}" "..."` command string
  - Remove all `.replace(/'/g, "''")` escaping
  - Replace with `prepare(sql).run/get/all(...params)` using `?` placeholders
  - Convert from `new Promise((resolve) => { exec(...) })` to direct synchronous better-sqlite3 calls wrapped in try/catch
  - Return the same response shape ({ success, settings/error })

  For `notification-settings:set` (the complex UPDATE/INSERT):
  - Build parameterized SET clauses: `setParts.push('field = ?')` and collect `params.push(value)`
  - For UPDATE: `prepare('UPDATE conversation_notification_settings SET ' + setParts.join(', ') + ' WHERE session_key = ?').run(...params, sessionKey)`
  - For INSERT: Use `prepare('INSERT INTO ... VALUES (?, ?, ?, ...)').run(sessionKey, ...values)`
  - Do NOT use string interpolation for any user-provided values

  IMPORTANT:
  - Do NOT change the IPC channel names
  - Do NOT change the response format returned to the renderer
  - Do NOT touch already-migrated mute/unmute handlers
  - The `prepare()` function caches statements, so the SQL string must be stable (no dynamic column names in the cache key -- use the dynamic SET clause approach shown above where the SQL is built from field names, which are code-controlled, not user-controlled)
  </action>
