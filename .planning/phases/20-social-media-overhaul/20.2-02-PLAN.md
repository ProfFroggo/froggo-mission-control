# Plan: Campaigns View Mode + Inline Approval Actions

## Objective

Add Campaigns as a fourth view mode in Pipeline. Add inline approval actions (approve/reject/schedule) directly on pipeline cards — replacing the separate approval pane.

## Execution Context

@src/components/XPipelineView.tsx
@src/components/XCampaignView.tsx (606 lines — campaign CRUD + scheduling)
@src/lib/api.ts (approvalApi, scheduleApi)

## Tasks

### Task 1: Campaigns as Pipeline view mode

**Requirements:**
- Add 'campaigns' to viewMode: `'board' | 'calendar' | 'list' | 'campaigns'`
- Fourth toggle button with Rocket icon
- When active, render campaign management UI (from XCampaignView):
  - Campaign list with cards (title, subject, stage count, status)
  - Campaign editor (title, subject, stages with day/time/content)
  - AI proposal banner (listens for `x-campaign-proposal` event)
  - Schedule campaign → creates per-stage approval requests
- Lift the campaign component logic inline or render XCampaignView as a sub-component
- Design polish: consistent card styling with pipeline cards (`rounded-xl border border-mission-control-border bg-mission-control-surface`)

### Task 2: Inline approval actions on pipeline cards

**Requirements:**
- Pipeline cards in "In Review" column show [Approve] [Reject] buttons directly
- Approve: calls `scheduleApi.update(id, { status: 'approved' })` + refreshes
- Reject: prompts for optional reason, calls `scheduleApi.update(id, { status: 'draft', metadata: { rejectionReason } })`
- Cards in "Approved" column show [Schedule] button with date picker
- Cards in "Scheduled" column show [Publish Now] button
- All actions already exist in PipelineCard — just ensure they're prominent and well-styled
- Action buttons: `text-xs px-3 py-1 rounded-lg` with semantic colors (green=approve, red=reject, blue=schedule)
- Remove the "Post Now" direct-post bypass from XReplyGuyView (line ~196) — all posts go through approval

### Task 3: Remove obsolete tab components from routing

**Requirements:**
- XContentEditorPane.tsx: remove cases for 'drafts', 'plan', 'calendar', 'publish' (now part of Pipeline or Compose modal)
- If XContentEditorPane becomes trivial (just routing to 4-5 components), inline the routing in XSocialLayout
- Update any custom event dispatches that reference removed tabs (e.g., `x-tab-change` with detail='publish' → open compose modal instead)
